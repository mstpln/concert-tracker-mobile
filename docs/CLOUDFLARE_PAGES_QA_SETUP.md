# Cloudflare Pages QA setup

## Purpose

The Pages project is a public synthetic preview only. It must never connect to the production Worker, production R2 data, provider APIs or user-owned ticket files. The generated build is expected to show the visible `QA PREVIEW · SYNTHETIC DATA` banner and to use only the fixtures committed under `qa/`.

## Create the Pages project

In the existing Cloudflare account open **Workers & Pages → Create application → Pages → Connect to Git**. Authorize GitHub and select `mstpln/concert-tracker-mobile`.

Use these settings:

- Project name: `livevault-qa`
- Production branch: `main`
- Framework preset: **None**
- Root directory: repository root
- Build command: `npm ci && npm run build:qa`
- Build output directory: `dist`
- Node version: 20

Do not add environment variables, secrets, Worker URLs, R2 bindings, provider keys, a production custom domain or production data. The build must remain deterministic from repository contents alone.

## Preview behavior

Pull requests should receive isolated preview URLs from Cloudflare Pages Git integration. A valid preview must:

- load the QA banner and reset control;
- use fictional bands, concerts, alerts, tickets and API usage;
- make no external provider or production-data requests;
- include `robots.txt`, `X-Robots-Tag` and the restrictive `_headers` policy generated by `build:qa`;
- use a QA-specific service-worker cache name containing the app version and build identifier.

The preview URL may be public because it contains only synthetic data. `noindex` reduces accidental indexing but is not access control, so the synthetic-data boundary remains mandatory.

## First manual verification

After the first successful Pages build:

1. Open the preview in a private browser window.
2. Confirm the QA banner is visible.
3. Navigate through Concerts, Dates, Bands and Alerts.
4. Change one checklist item, reload and confirm it persists.
5. Use **Reset QA Data** and confirm the fixture state returns.
6. Check the browser network panel for requests only to the preview origin and the local synthetic endpoint.
7. Confirm no production connection form values, personal concert history or ticket files appear.

## Troubleshooting

If the build fails, verify Node 20, the committed npm lockfile, the exact build command and the `dist` output directory. If the preview opens but the app shell is stale, delete the preview site data and confirm the generated service-worker cache name includes the current QA build identifier. If external requests appear, stop review and treat that as a release blocker.

Creating or changing the Cloudflare Pages project is a manual Cloudflare action. Merging a pull request does not deploy the production Worker or modify production R2 data.
